About pd-vm and RustScript

Overview

The playground is a browser UI for RustScript built on top of pd-vm running inside the wasm runtime in your browser, with Monaco as the editor surface.

The current shell is intentionally RustScript-focused: the main toolbar exposes bundled RSS examples, formatting, linting, run/debug controls, and interruption controls around the RustScript workflow.

What The Playground Provides

  • Live lint diagnostics surfaced as Monaco markers and a diagnostics panel.
  • One-click document formatting for the RustScript editor.
  • Completion entries and callable hover docs sourced from the wasm completion catalog.
  • Compiler-driven inferred local type hover for visible identifiers, even outside an active debug session.
  • Run and debug workflows directly in the browser.
  • Breakpoints, step/next/out/continue, stack and locals views, and hover inspection in the debugger.
  • Print output and final stack panels.
  • An RSS Program picker that loads bundled RustScript examples and tracks non-bundled edits as a Custom Draft.
  • Theme preference and editor source persistence in browser storage.
  • Fuel and epoch interruption controls, plus live interruption state shown in the toolbar.

Bundled RustScript Samples

The RustScript program picker currently ships with:

  • Demo: closures, structs, optional matching, bytes, JSON, regex, and runtime::sleep.
  • IFFT Example: array-heavy numeric code and validation.
  • LRU Cache Example: typed structs plus stdlib lrucache helpers.
  • Collections + Iter: stdlib collections and iterator helpers.
  • Strings + Regex: string transforms and regex parsing.

Interruption Controls

The playground starts with interruption disabled by default. You can explicitly choose Fuel or Epoch in the Interruption panel before starting a run or debug session.

Fuel

Fuel mode yields execution when the active session runs out of fuel. In the browser UI, fuel is configured per session together with a check interval.

Epoch

Epoch mode uses a browser-side JavaScript timer. In the playground, one epoch tick is driven by a 1ms timer when ticking is enabled. The UI shows current epoch, deadline, slice, and check interval.

The epoch toggle button pauses or resumes that timer. This is a browser embedding detail, not a wall-clock guarantee from the VM itself.

Async Host Waiting In The Browser

In the wasm playground, runtime::sleep(ms) is not a blocking native sleep and it is not an immediate no-op. It is implemented as a browser-side async host operation: the VM yields, the host bridge tracks a deadline with browser time, and the run/debug session resumes once that deadline is reached.

Debugger

The debugger supports:

  • Line breakpoints in the Monaco gutter.
  • where, locals, and stack inspection.
  • step, next, out, continue, and stop controls.
  • Hover inspection for visible variables when a debug session is active.

RustScript In The Playground

Supported RustScript Syntax And Features

  • Statements: use, fn, pub fn, let, assignment, indexed/member assignment, if/else, while, C-style for, break, continue.
  • Expressions: literals (int, float, bool, string, null), arithmetic, logical, comparison, function calls, closures (|...| expr), if-expression form (if cond => { ... } else => { ... }), match expressions.
  • Match patterns: int/string/null literals, wildcard _, None, non-null binding patterns Some(name), and type constructors Some(TypeName) / Option::Some(TypeName).
  • Collections: array literal [], brace literals for arrays/maps, obj.member, obj[key], optional chaining (?. and ?.[key]), slice syntax ([a:b], [:b], [a:]), map key literals including null.
  • Builtins and namespaces: bytes::..., io::..., re::..., json::..., math::..., and jit::....
  • Host namespaces: use runtime;, use runtime as alias;, and use runtime::{sleep as nap}; style imports are supported.
  • RustScript frontend rewrites: Option::None lowers to null, Option::Some(expr) lowers to the wrapped expression, and print/println support Rust-style formatting placeholders.
  • Optional-chain typing: in RustScript, ?. requires a user-declared schema. Optional results must be handled with .unwrap_or(...), a != null refinement, or a match arm that binds Some(name) before normal use, and handled values keep their concrete type in both compiler and lint diagnostics.
  • Module imports: use module;, use module::*;, use module::{name};, and relative paths via self:: / super::.

Ownership And Liveness Behavior

  • let bindings are immutable by default; use let mut for mutable bindings.
  • Reassignment, indexed/member mutation, and &mut borrows require a mutable binding.
  • Non-copyable field reads (e.g., p.a) and literal index element moves (e.g., let x = arr[0]) are move-by-default and clear the moved field/element on the owning container.
  • Non-copyable local-to-local rebinds (e.g., let b = a) move the source local by default.
  • RustScript function calls apply consumed-parameter inference; reusing a consumed caller local after the call is rejected.
  • Use .copy() to explicitly clone/detach a value.
  • &expr and &mut expr are accepted borrow forms. In the current subset, they act as non-consuming access forms but participate in move/liveness checks.
  • Collection aliases are tracked through assignment, borrow binding, and passthrough calls; mutating a collection while an alias exists is rejected unless detached (e.g., via .copy()).
  • Locals must be definitely available on every control-flow path before use.
  • Closures and fn declarations capture outer locals at definition/declaration time. Capture mode follows expression semantics (x can move, x.copy() copies, and &x / &mut x keep borrow-style capture behavior).
  • Inline fn and closure calls clear per-call frame slots after producing the call result. Closure capture slots participate in liveness null-clears once the owning closure/value is dead.

Current RustScript Compiler Limitations

  • crate::... module paths are not supported in source loading; use relative module paths.
  • Callable locals can be passed and called, but are not runtime Values and cannot be stored in arrays/maps or returned.
  • Recursive RustScript function declarations are not supported by current inlining-based lowering.
  • Function declarations implicitly capture outer locals (snapshot at declaration time).
  • In move-semantics mode, implicit captures follow expression semantics.
  • match patterns are limited to int/string/null literals, None, Some(name), _, and type constructors (Some(TypeName)).
  • break and continue are only valid inside loops.
  • Host import namespace allows runtime::, and builtin namespaces include bytes::, io::, re::, json::, math::, and jit::.

Browser Runtime Limitations

  • The playground uses the wasm runtime. Native JIT is not available here.
  • Execution in the browser is effectively interpreter-only today.
  • Many host io::... operations are unavailable on wasm, including file and process-backed I/O such as io::open, io::popen, and related handle operations.
  • jit::... configuration helpers exist in the language/runtime surface, but they do not enable native trace compilation inside the browser build.
  • Epoch mode depends on browser timer delivery. A long compute-only wasm call cannot be preempted mid-call while the main thread is busy.
  • Browser behavior can differ from native CLI or server embeddings where threads, timers, and native execution paths are available.
